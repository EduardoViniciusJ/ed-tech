## STRY0030 - Definir UI Policies nos Record Producers Financeiros

**Prioridade**: Alta  
**Update Set**: EDTECH_03_FINANCEIRO_STRY0030_UI_POLICY_RPS_FINANCEIROS  
**Parent**: EDTECH_03_FINANCEIRO

**Persona**:  
Aluno que precisa abrir solicitações financeiras pelo portal preenchendo apenas os dados necessários.

**O que precisa**:  
Criar **Catalog UI Policies** para controlar campos visíveis, obrigatórios e somente leitura nos Record Producers financeiros.

**Record Producers impactados**:

- Solicitar 2ª Via de Boleto
- Enviar Comprovante de Pagamento
- Contestar Multa por Atraso de Boleto

**Campos somente leitura**:

- Solicitante
- RA/Matrícula
- Curso
- Campus
- E-mail de contato
- Telefone, se vier do cadastro

**Regras por formulário**:

**Solicitar 2ª Via de Boleto**

- Mostrar: Mês/competência, data de vencimento, número do boleto, observações, assunto curto e descrição detalhada.
- Obrigatório: Mês/competência, data de vencimento e assunto curto.
- Ocultar: Comprovante de pagamento e campos de contestação de multa.

**Enviar Comprovante de Pagamento**

- Mostrar: Mês/competência, valor aproximado, número do boleto, comprovante, observações, assunto curto e descrição detalhada.
- Obrigatório: Mês/competência, comprovante de pagamento e assunto curto.
- Ocultar: Campos de contestação de multa.

**Contestar Multa por Atraso de Boleto**

- Mostrar: Mês/competência, data de vencimento, valor aproximado, número do boleto, justificativa/descrição, anexo/evidência, assunto curto e descrição detalhada.
- Obrigatório: Mês/competência, data de vencimento, valor aproximado, justificativa/descrição e assunto curto.

**O que fazer no ServiceNow**:

- Criar uma **Catalog UI Policy** para cada RP financeiro.
- Nomes sugeridos:
    - UP - RP Financeiro - 2 Via de Boleto
    - UP - RP Financeiro - Enviar Comprovante
    - UP - RP Financeiro - Contestar Multa
- Configurar Visible, Mandatory e Read only nas UI Policy Actions.
- Testar os três formulários no Portal/Employee Center com usuário aluno.

**Critérios de aceite**:

- Cada RP mostra apenas os campos necessários para seu tipo de solicitação.
- Dados do aluno ficam somente leitura quando vierem do cadastro.
- Comprovante é obrigatório no RP de envio de pagamento.
- Contestação de multa exige dados suficientes para análise.
- Nenhum campo local duplica campo já existente em Variable Set.

**O que precisa**:  
Atualizar o campo Script dos Record Producers da Secretaria para copiar as variáveis dos Variable Sets para os campos reais da tabela u_edu_secretaria.

- Emitir Comprovante de Matrícula

`current.u_tipo_documento = producer.tipo_documento; current.u_forma_entrega = producer.forma_entrega; current.u_finalidade_documento = producer.finalidade_documento; current.u_observacoes_adicionais = producer.observacoes_adicionais;`

## STRY0037 - Criar Assignment Rule para Solicitações da Secretaria

**Prioridade**: Alta  
**Update Set**: EDTECH_04_SECRETARIA_STRY0037_ASSIGNMENT_RULE_SECRETARIA  
**Parent**: EDTECH_04_SECRETARIA

**O que precisa**:  
Criar uma **Assignment Rule** para atribuir automaticamente as solicitações da tabela u_edu_secretaria ao grupo Secretaria.

**Onde aplicar**:

- Tabela Solicitação Secretaria [u_edu_secretaria]
- Grupo Secretaria

**O que fazer no ServiceNow**:

- Abrir System Policy > Rules > Assignment.
- Criar Assignment Rule AR - Secretaria - Atribuir Solicitações ao Grupo Secretaria.
- Table: Solicitação Secretaria [u_edu_secretaria].
- Active: true.
- Condições:
    - Assignment group is empty
    - Active is true
- Aba Assign To:
    - Grupo: Secretaria
- Não definir assignment_group no script dos Record Producers.

**Critérios de aceite**:

- Toda solicitação criada em u_edu_secretaria cai no grupo Secretaria.
- A regra não altera registros que já tenham assignment_group preenchido.
- A Secretaria visualiza as solicitações na fila do grupo.
- O script dos RPs não possui grupo hardcoded.

Você está certo em desconfiar: Suporte usa Incident [incident], então não precisa seguir exatamente a mesma lógica das tabelas custom u_edu_secretaria e u_edu_financeiro.

## STRY0040 - Criar SLA de 3 Dias Úteis para Solicitações da Secretaria

**Prioridade**: Alta  
**Update Set**: EDTECH_04_SECRETARIA_STRY0040_SLA_SECRETARIA_3_DIAS  
**Parent**: EDTECH_04_SECRETARIA

**Precisa da STRY0037 para as solicitações da Secretaria caírem no grupo Secretaria e da STRY0039 para o aluno ser notificado sobre o andamento.**

**O que precisa**:  
Criar controle de prazo/SLA para garantir que solicitações da Secretaria sejam atendidas em até **3 dias úteis**, conforme o problema apresentado no TCC.

**Onde aplicar**:

- Tabela Solicitação Secretaria [u_edu_secretaria]
- Grupo Secretaria
- Record Producers:
    - Emitir Comprovante de Matrícula
    - Solicitar Declaração Acadêmica
    - Solicitar Histórico Escolar
    - Solicitar Diploma

**Regra do SLA**:

- Iniciar quando a solicitação for criada em u_edu_secretaria.
- Prazo alvo: 3 dias úteis.
- Pausar quando state = Pending, pois depende de resposta do aluno.
- Retomar quando sair de Pending.
- Encerrar quando state = Closed Complete, Closed Incomplete ou Closed Skipped.

**O que fazer no ServiceNow**:

- Abrir Service Level Management > SLA Definitions.
- Criar SLA SLA - Secretaria - Documentos Acadêmicos - 3 Dias Úteis.
- Table: u_edu_secretaria.
- Duration: 3 business days.
- Start condition:
    - Active is true
    - Assignment group is Secretaria ou tabela u_edu_secretaria, conforme configuração disponível.
- Pause condition:
    - State is Pending
- Stop condition:
    - State is Closed Complete
    - State is Closed Incomplete
    - State is Closed Skipped
- Validar calendário/horário comercial usado pelo SLA.
- Testar com solicitação criada pelo portal.

**Critérios de aceite**:

- Solicitações da Secretaria possuem SLA de 3 dias úteis.
- SLA inicia automaticamente após criação/atribuição da solicitação.
- SLA pausa quando a solicitação fica aguardando aluno.
- SLA encerra quando a solicitação é fechada.
- Solicitações próximas do vencimento ou vencidas ficam identificáveis para a equipe/gestor.
- O comportamento funciona para os 4 Record Producers da Secretaria.
