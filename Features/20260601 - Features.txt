## STRY0096 - Criar Role Operacional de Atendimento para Secretaria e Financeiro

**Prioridade**: Alta  
**Update Set**: `EDTECH_01_CORE_V2_STRY0096_ROLE_ATENDIMENTO_OPERADOR`  
**Parent**: `EDTECH_01_CORE_V2`

**Persona**: Secretaria Acadêmica e Financeiro

**O que precisa**:  
Permitir que usuários internos da Secretaria e do Financeiro atualizem campos operacionais de atendimento nas solicitações, sem receber perfil `admin`, sem depender de `itil` amplo e sem permissão de exclusão.

**Escopo desta story**:

- Criar ou validar a role `u_edu_atendimento_operador`.
- Vincular a role aos grupos `Secretaria` e `Financeira`.
- Liberar escrita apenas nos campos operacionais necessários.
- Manter dados do aluno, dados acadêmicos e detalhes originais da solicitação somente leitura.
- Manter `delete` restrito a `admin`.
- Não criar Catalog Item, Record Producer ou página de portal.

**Onde aplicar**:

- User Administration > Roles
- User Administration > Groups
- System Security > Access Control (ACL)
- Tabelas `u_edu_secretaria`, `u_edu_financeiro` e campos herdados de `Task [task]`

**O que fazer no ServiceNow**:

- Criar ou validar a role `u_edu_atendimento_operador`.
- Adicionar `u_edu_atendimento_operador` ao grupo `Secretaria`.
- Adicionar `u_edu_atendimento_operador` ao grupo `Financeira`.
- Criar ou validar ACL de `write` para `u_edu_secretaria`, usando `u_edu_atendimento_operador`.
- Criar ou validar ACL de `write` para `u_edu_financeiro`, usando `u_edu_atendimento_operador`.
- Criar ACLs de campo em `Task [task]` para campos herdados usados no atendimento:
    - `task.state` com operation `write`
    - `task.priority` com operation `write`
    - `task.assignment_group` com operation `write`, se o grupo responsável puder ser ajustado manualmente
    - `task.work_notes` com operations `read` e `write`
    - `task.comments` com operation `write`, se comentários visíveis ao solicitante forem usados
- Nas ACLs de `Task [task]`, usar script para limitar o acesso apenas às classes do projeto:

```javascript
answer = current.sys_class_name == 'u_edu_secretaria' || current.sys_class_name == 'u_edu_financeiro';
```

- Não conceder `delete` para `u_edu_atendimento_operador`.
- Não conceder `itil` apenas para resolver permissão de atendimento, salvo decisão consciente para ambiente de demonstração.

**Critérios de aceite**:

- Usuário da Secretaria consegue alterar estado, prioridade, notas internas, comentários e fechamento em solicitações da Secretaria.
- Usuário do Financeiro consegue alterar estado, prioridade, notas internas, comentários e fechamento em solicitações financeiras.
- Usuário da Secretaria não consegue excluir solicitações.
- Usuário do Financeiro não consegue excluir solicitações.
- A role não libera edição geral em `incident`, `sc_task`, `change_request` ou outras tabelas que estendem `Task`.
- Campos herdados de `Task` deixam de apresentar erro `Security prevents writing to this field` nas tabelas permitidas.

## STRY0099 - Criar Notificações In-App e Revisar E-mails da Secretaria

**Prioridade**: Alta  
**Update Set**: `EDTECH_04_SECRETARIA_V2_STRY0099_NOTIFICACOES_SECRETARIA_IN_APP`  
**Parent**: `EDTECH_04_SECRETARIA_V2`

**Persona**: Aluno e Secretaria Acadêmica

**O que precisa**:  
Garantir que o aluno e a Secretaria sejam notificados corretamente sobre abertura e atualização de solicitações, tanto por e-mail quanto por notificação interna no ServiceNow quando suportado pela instância.

**Escopo desta story**:

- Revisar destinatários das Email Notifications existentes da Secretaria.
- Criar notificação para a equipe Secretaria quando uma nova solicitação for aberta.
- Criar notificações internas/in-app para aparecer no sino do ServiceNow, se o recurso estiver disponível.
- Manter notificações por solicitação individual, sem enviar para todos os alunos.
- Não criar novo portal ou widget nesta story, salvo se for necessário para exibir notificações no portal customizado.

**Onde aplicar**:

- System Notification > Email > Notifications
- System Notification > Provider > Notifications, se disponível na instância
- Flow Designer, se a ação de in-app notification estiver disponível
- Tabela `u_edu_secretaria`
- Campo `u_solicitante`
- Grupo `Secretaria`

**O que fazer no ServiceNow**:

- Revisar Email Notifications de Secretaria:
    - chamado aberto
    - chamado em andamento
    - chamado pendente
    - chamado concluído
    - chamado atualizado
- Em notificações para o aluno, usar destinatário:
    - `Users/Groups in fields` = `Solicitante`
- Remover grupo `Alunos` de notificações individuais.
- Criar notificação de e-mail para equipe interna:
    - Nome: `SEC - Nova solicitação para Secretaria`
    - Tabela: `u_edu_secretaria`
    - Quando enviar: registro inserido
    - Destinatário: grupo `Secretaria`
- Criar Provider Notification ou notificação in-app para equipe Secretaria:
    - Nome: `SEC - Nova solicitação para Secretaria - In-app`
    - Tabela: `u_edu_secretaria`
    - Disparo: registro inserido
    - Destinatário: grupo responsável ou grupo `Secretaria`
    - Mensagem: `Nova solicitação acadêmica aberta: ${number}`
- Criar Provider Notification ou notificação in-app para aluno:
    - Nome: `SEC - Solicitação aberta para aluno - In-app`
    - Tabela: `u_edu_secretaria`
    - Disparo: registro inserido
    - Destinatário: `Solicitante`
    - Mensagem: `Sua solicitação ${number} foi aberta com sucesso.`
- Criar notificação in-app de atualização para aluno:
    - Disparo: registro atualizado
    - Condição: estado da solicitação muda
    - Destinatário: `Solicitante`
    - Mensagem: `Sua solicitação ${number} foi atualizada.`
- Validar se o portal customizado possui header/widget que exibe notificações internas.
- Se o portal não exibir notificações in-app, registrar isso como limitação técnica e manter e-mail/sys_email como validação do MVP.

**Critérios de aceite**:

- Ao abrir solicitação da Secretaria, o aluno recebe notificação individual.
- Ao abrir solicitação da Secretaria, o grupo Secretaria recebe aviso da nova demanda.
- Notificações individuais não são enviadas para todos os usuários do grupo Alunos.
- Mudança de estado gera notificação para o solicitante.
- Registros aparecem em `sys_email.list` quando a notificação de e-mail dispara.
- Notificações internas aparecem no sino do ServiceNow quando o recurso de Provider Notification estiver disponível.
- O comportamento e as limitações da PDI ficam documentados para apresentação.
